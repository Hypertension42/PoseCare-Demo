# PoseCare Demo 架构

## 1. 一句话定位

PoseCare 是一个医学健康方向的视觉搜索 Web Demo。用户不用搜索“正确坐姿”或“深蹲教程”，只要拍下自己的姿势并直接提问，系统就能在画面里标出姿势问题，解释风险，并给出纠正建议。

## 2. 比赛里的产品判断

赛道二看的是“视觉 + 语音，所见即所想”，所以我们不能做成普通问答 Agent，也不能只做一个姿势识别小工具。

最终形态应该是：

> Web/H5 产品界面 + 姿势识别引擎 + AI 姿势分析 Agent + 可分享报告卡

其中：

- Web/H5 是主体产品，评委能直接打开体验。
- 摄像头画面是第一入口，保证“视觉搜索”的感觉。
- 语音/文字问题是第二入口，保证“所见即所想”。
- Agent 负责解释和追问，不作为主界面。
- 报告卡负责传播力和摆摊展示。

## 3. MVP 范围

第一版只做一个主场景，优先级如下：

1. 坐姿检测：最稳定、最容易现场演示，也最贴近日常。
2. 深蹲检测：视觉冲击强，但对视频和角度要求更高。
3. 站姿体态：展示好看，但医学解释容易变宽。
4. 康复动作：适合讲未来扩展，第一版不建议作为主 Demo。

建议第一版主打：

> 久坐姿势检测：头前伸、圆肩、驼背、腰背弯曲、高低肩

只要能稳定展示“人体骨架 + 风险点 + 纠正建议 + 报告卡”，Demo 就成立。

## 4. 用户旅程

### 4.1 捕捉

用户打开网页后直接进入相机界面。

可用方式：

- 现场摄像头拍摄
- 上传图片
- 上传 5-10 秒短视频
- 使用内置样例图，保证比赛现场兜底

### 4.2 提问

用户可以问：

- “我这个坐姿哪里不对？”
- “我是不是有点驼背？”
- “这样坐会不会伤腰？”
- “怎么调整？”

MVP 里语音可以做成“按住说话”的交互；如果浏览器语音识别不稳定，就保留文字输入作为兜底。

### 4.3 理解

系统完成三件事：

- 从画面识别人体关键点。
- 计算姿态特征，例如肩膀高度、躯干倾斜、颈部前伸、髋膝踝对齐。
- 结合用户问题生成可理解的健康建议。

### 4.4 展示

结果页必须有视觉冲击：

- 原图/视频上叠加人体骨架。
- 问题部位标红或标黄。
- 展示风险等级：低风险 / 中风险 / 高风险。
- 展示 2-3 条纠正建议。
- 支持生成“姿势体检卡”。

## 5. 系统架构

```text
用户
  |
  v
Web/H5 前端
  |
  +-- 相机/上传模块
  |
  +-- 语音/文字提问模块
  |
  +-- 姿态识别模块
  |     |
  |     +-- MediaPipe Pose Landmarker 或 TensorFlow.js MoveNet
  |     +-- 输出人体关键点
  |
  +-- 姿势风险规则模块
  |     |
  |     +-- 关节角度计算
  |     +-- 左右对称性计算
  |     +-- 场景化风险评分
  |
  +-- AI 姿势分析 Agent
  |     |
  |     +-- 输入：用户问题 + 姿态指标 + 场景
  |     +-- 输出：解释、建议、追问、风险提示
  |
  +-- 结果渲染模块
        |
        +-- 骨架叠加
        +-- 风险点高亮
        +-- 建议面板
        +-- 报告卡生成
```

## 6. 技术选型

### 6.1 前端

建议：

- Vite + React + TypeScript
- 普通 CSS 或 Tailwind CSS
- Canvas/SVG 叠加人体骨架
- html-to-image 或 html2canvas 生成报告卡

理由：

- 启动快。
- 适合移动端 H5。
- 容易做视觉层和交互层。
- 比赛现场部署简单。

### 6.2 姿态识别

优先选择：

- MediaPipe Pose Landmarker

备选：

- TensorFlow.js MoveNet

判断：

- MediaPipe Pose Landmarker 更适合人体姿态 Demo，关键点更丰富，适合做肩、髋、膝等分析。
- MoveNet 更轻，17 个关键点，适合快速做单人骨架和动作检测。

第一版建议从 MediaPipe 开始。如果接入太慢，立刻降级为 MoveNet 或静态样例数据，不要卡在模型接入上。

官方参考：

- MediaPipe Pose Landmarker Web: https://ai.google.dev/edge/mediapipe/solutions/vision/pose_landmarker/web_js
- TensorFlow.js MoveNet: https://www.tensorflow.org/hub/tutorials/movenet

### 6.3 Agent / 大模型

Agent 不直接看原始视频。第一版更稳的做法是：

```text
画面 -> 姿态关键点 -> 姿态指标 JSON -> Agent -> 中文解释
```

输入示例：

```json
{
  "scene": "sitting",
  "question": "我这个坐姿哪里不对？",
  "riskLevel": "medium",
  "findings": [
    { "part": "neck", "issue": "head_forward", "severity": 0.72 },
    { "part": "shoulder", "issue": "rounded_shoulder", "severity": 0.55 },
    { "part": "back", "issue": "trunk_flexion", "severity": 0.61 }
  ]
}
```

Agent 输出必须结构化：

```json
{
  "title": "中等姿势风险",
  "summary": "你的坐姿存在明显头前伸和腰背弯曲。",
  "explanation": "长时间保持这种姿势可能增加颈肩和腰背负担。",
  "suggestions": [
    "把屏幕上沿调整到接近眼平高度。",
    "肩膀放松，胸廓轻微打开。",
    "每 30-45 分钟起身活动 2 分钟。"
  ],
  "medicalBoundary": "本结果仅用于姿势风险筛查，不能替代医生诊断。"
}
```

这样做的好处是 Agent 负责表达和推理，不负责不稳定的底层视觉检测。

## 7. 姿势风险规则

第一版不要追求医学诊断，做“姿势风险筛查”即可。

### 7.1 坐姿规则

可检测项：

- 头前伸：耳朵/鼻子相对肩膀明显前移。
- 高低肩：左右肩关键点高度差过大。
- 驼背/躯干前倾：肩、髋连线与垂直方向夹角过大。
- 腰背弯曲：躯干角度异常，结合用户坐姿场景判断。
- 久坐风险：根据用户自报时长或默认提示。

输出：

- 低风险：只有轻微偏移。
- 中风险：出现 1-2 个明显问题。
- 高风险：多个问题同时出现，或用户提到疼痛/麻木等症状。

### 7.2 深蹲规则

可检测项：

- 膝盖内扣：膝盖点相对髋-踝连线向内偏移。
- 躯干过度前倾：肩髋连线角度过大。
- 左右不对称：左右膝、髋、踝角度差异明显。
- 膝盖轨迹不稳定：视频帧之间抖动或偏移明显。

第一版可作为第二个展示场景，不作为主交付。

## 8. 页面结构

### 8.1 相机页

目标：让用户一眼知道这是“拍一下就能问”的产品。

主要元素：

- 实时相机区域
- 场景切换：坐姿 / 站姿 / 深蹲
- 拍照按钮
- 上传按钮
- 语音提问按钮
- 文字输入兜底

### 8.2 分析页

目标：展示视觉理解能力。

主要元素：

- 原图/视频画面
- 人体骨架叠加
- 风险部位高亮
- 风险分数或等级
- 核心问题列表

### 8.3 建议页

目标：展示医学健康价值。

主要元素：

- 简短解释
- 纠正建议
- 复查提醒
- 医疗边界说明

### 8.4 报告卡页

目标：传播和比赛展示。

主要元素：

- 用户姿势截图
- 风险等级
- 3 个主要发现
- 3 条纠正建议
- 项目名和二维码/口号

报告卡可以叫：

> 姿势体检卡

## 9. 数据流

```text
Camera / Upload
  -> Frame Capture
  -> Pose Estimation
  -> Landmark Normalization
  -> Feature Extraction
  -> Risk Scoring
  -> Agent Explanation
  -> Result UI
  -> Share Card
```

关键点：

- 姿态识别尽量在浏览器本地跑，减少网络不稳定风险。
- 大模型只处理结构化姿态结果和用户问题。
- 如果大模型接口不可用，使用本地模板生成解释，Demo 仍然可跑。

## 10. 项目目录建议

```text
PoseCare-Demo/
  docs/
    ARCHITECTURE.md
  src/
    app/
    features/
      capture/
      pose/
      analysis/
      agent/
      report-card/
    data/
      demo-samples/
    styles/
```

模块职责：

- capture：摄像头、上传、截图、视频帧抽取。
- pose：接入姿态识别模型，输出关键点。
- analysis：角度、对称性、风险等级和规则。
- agent：把结构化结果转成自然语言建议。
- report-card：生成可分享卡片。

## 11. Demo 兜底策略

比赛 Demo 最怕现场识别不稳，所以必须设计兜底。

兜底顺序：

1. 实时摄像头识别。
2. 上传现场拍好的照片。
3. 使用内置样例图。
4. 使用预置关键点 JSON 回放完整流程。

即使模型当天失效，展示链路也要能跑完：

```text
样例图 -> 预置骨架 -> 预置风险结果 -> Agent 文案 -> 报告卡
```

## 12. 医学边界

产品定位必须是：

> 姿势风险筛查和康复动作辅助反馈，不是疾病诊断。

不能说：

- “诊断颈椎病”
- “判断腰椎间盘突出”
- “诊断脊柱侧弯”

可以说：

- “提示颈肩负担风险”
- “提示腰背姿势压力”
- “提示动作代偿风险”
- “建议有疼痛、麻木、无力等症状时就医”

## 13. 评分点映射

对应赛道二评分：

- 场景与问题洞察：久坐、健身和康复训练都是真实高频场景。
- AI 能力与产品结合：视觉识别人体姿态，Agent 解释风险和建议。
- 体验完整性：拍摄、提问、分析、追问、报告卡，形成完整旅程。
- 用户价值感：用户得到的是“我的姿势哪里不对”，不是泛泛教程。
- 创新性与延展潜力：可以扩展到健身、康复、工位人体工学、老人护理。

## 14. 开发阶段

### 阶段 1：静态可演示闭环

- 做移动端页面。
- 放一张样例坐姿图。
- 手动画骨架和风险点。
- 输出固定分析结果。
- 生成报告卡。

目标：半天内拿到一个能讲故事的 Demo。

### 阶段 2：接入真实姿态识别

- 接入 MediaPipe 或 MoveNet。
- 从图片/摄像头中识别关键点。
- 用 Canvas 画骨架。
- 根据规则输出风险等级。

目标：让 Demo 从“演示稿”变成“真实可体验”。

### 阶段 3：接入语音和 Agent

- 语音转文字。
- 把用户问题和姿态指标传给 Agent。
- Agent 返回结构化建议。
- 无接口时走本地模板。

目标：体现“视觉 + 语音，所见即所想”。

### 阶段 4：比赛打磨

- 移动端适配。
- 展示动效。
- 内置样例。
- 准备答辩演示脚本。
- 准备网络不可用兜底。

## 15. 当前决策

当前先按这个方向推进：

- 产品形态：Web/H5。
- 主演示场景：坐姿检测。
- 技术路线：浏览器本地姿态识别 + 规则评分 + Agent 解释。
- 输出物：姿势体检卡。
- 医学边界：风险筛查，不做疾病诊断。
